# Operator console localization (i18n)

This document describes the lightweight runtime localization layer added on
2026-06-17 and how to extend it. It is intentionally small: the goal was to make
the operator console **localization-ready** without an Angular/Material build
migration or a heavy new dependency.

> Scope note: this PR delivers the infrastructure plus the English source
> catalog. Replacing inline template strings with catalog lookups is an
> incremental follow-up — see [Rollout order](#rollout-order).

## Pieces

| File | Responsibility |
| ---- | -------------- |
| `web/src/app/services/i18n.messages.en.ts` | English **source of truth** catalog (`EN_MESSAGES`) + `MessageKey` / `LocaleCatalog` types. |
| `web/src/app/services/i18n.service.ts` | `I18nService`: active-locale signal, catalog registry, `translate()`, `plural()`. |
| `web/src/app/i18n/translate.pipe.ts` | Standalone `translate` pipe for templates. |

## Principles

- **English is the source.** Every key has an English value. Other locales are
  registered at runtime and may be **partial** — any missing key falls back to
  English, and an unknown key falls back to the key string itself, so the UI
  never renders blank.
- **No build migration.** This is a runtime registry, not `@angular/localize`
  message extraction. It can coexist with a future compile-time approach if one
  is ever adopted.
- **Stable interpolation & pluralization.** Externalized strings keep the same
  runtime behaviour they had inline.

## Externalizing a string

1. Add a namespaced key to `EN_MESSAGES`:

   ```ts
   "home.systemChecks": "System checks",
   ```

2. Use it in a template via the pipe:

   ```html
   <h2> 'home.systemChecks' | translate </h2>
   ```

3. With interpolation params:

   ```ts
   // EN_MESSAGES
   "home.rerunCheck": "Re-run check: {label}",
   ```

   ```html
   <button [attr.aria-label]="'home.rerunCheck' | translate:{ label: check.label }">
   ```

4. In component code, inject the service:

   ```ts
   private readonly i18n = inject(I18nService)
   // ...
   this.i18n.translate("state.connected")
   ```

## Pluralization

Define sibling `.one` / `.other` keys and resolve with `plural()`, which always
exposes the count as `{count}`:

```ts
// EN_MESSAGES
"home.checksNeedReview.one": "{count} check needs review",
"home.checksNeedReview.other": "{count} checks need review",
```

```ts
this.i18n.plural("home.checksNeedReview", failing.length)
// 1 -> "1 check needs review"
// 4 -> "4 checks need review"
```

## Adding a locale

```ts
import { I18nService } from "./services/i18n.service"

const i18n = inject(I18nService)
i18n.registerCatalog("fil", {
  "nav.home": "Tahanan",
  "nav.settings": "Mga Setting",
  // partial is fine; missing keys fall back to English
})
i18n.setLocale("fil")
```

## Rollout order

Externalize visible strings surface by surface so each step is reviewable and
low-risk:

1. Navigation (rail + bottom bar labels)
2. Top bar
3. Bottom navigation
4. Dashboard headings
5. Common actions (save / cancel / retry / close)
6. State & status messages
7. Shared dialogs
8. Consent / legal copy **only if** the precise legal meaning is preserved —
   coordinate with whoever owns the RA 10173 / terms wording before changing it.

## Translation-safe layout

Localized strings are frequently longer than English. When externalizing labels:

- Do **not** hard-code widths derived from English text length.
- Allow wrapping for low-emphasis labels with the shared `.r-label` helper
  (`white-space: normal; overflow-wrap: anywhere; hyphens: auto;`).
- Navigation tab labels already wrap to two lines (see
  `navigation-tab.component.ts`); verify new strings at the responsive
  breakpoints (<760, 760–1023, 1024–1219, 1220–1498, 1499+) and with safe-area
  insets so nothing introduces horizontal scroll.

## Testing

`web/src/app/services/i18n.service.spec.ts` covers resolution, fallback,
interpolation, pluralization, and locale switching. Run the web unit suite
(vitest) as configured in the repo.
